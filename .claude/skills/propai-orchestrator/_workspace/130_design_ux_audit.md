## 0. 핵심 요약 (Executive Summary)

PropAI는 **단일 디자인 토큰 시스템(`packages/ui/src/styles/tokens.css`)**, **단계 SSOT(`lib/lifecycle-stages.ts`)**, **표준 3구역 페이지 패턴(헤더+위젯+CTA)** 등 좋은 구조적 골격을 이미 갖췄다. 그러나 다음 4가지가 제품 완성도를 깎는다.

| # | 발견 | 심각도 | 근거 |
|---|------|--------|------|
| **A** | **진행 레일·다음단계 CTA가 라이브에서 전혀 렌더되지 않음** (가이디드 워크플로우 차별화 기능이 프로덕션에서 死상태) | **치명** | 라이브 `propai-project-context.projectId = null` (프로젝트 진입 중에도). 레일·CTA는 이 값에 게이트됨 |
| **B** | **토큰 우회 하드코딩 색상 수백 건** (다크 기본 모드에서 대비 붕괴 위험) | 높음 | `text-slate-900` 30건·`bg-emerald-500` 33건 등. globals.css 보정 레이어가 `bg-*`/`border-*`/`text-slate-900`은 못 잡음 |
| **C** | **단계 헤더(ModulePlaceholder)가 "준비중" 마케팅 블록처럼 보임** + 개발자 용어 노출(PROP/FINANCE, ko, 실연동, raw UUID) | 높음 | 라이브 finance/bim 화면. 실기능 페이지 상단을 따옴표 인용문·"범위 및 준비 상태"가 차지 |
| **D** | **버튼·탭 라벨 비표준화** (분석 실행 4종 표기·"로그인 실행"·단계 수 6 vs 10 불일치) | 중간 | grep 결과 + 라이브 카피 |

가장 임팩트 큰 단일 조치는 **A 수정**(이미 만들어둔 가이디드 UX를 켜는 것). 노력 대비 효과 최상.

## 1. 디자인 일관성

### 1.1 강점
- **토큰 체계가 견고하다.** `tokens.css`에 라이트/다크 양 모드, surface 4단계·text 4단계·status·chart 6색·shadow 7단계·radius 5단계·motion easing까지 정의. 다크가 기본(`<html className="dark">`).
- **다크 가독성 근본 보정 레이어**(globals.css 527~547)가 `:where(.dark)` 명시도 0 기법으로 하드코딩된 `text-slate-400/500/600`·`text-white/20~50`을 토큰 가독색으로 자동 상향 — 영리한 회귀-제로 보정.
- **고대비 모드**(`prefers-contrast: high`)·**모션 감소**(`prefers-reduced-motion`) 미디어쿼리 대응.
- **분양앱용 `@layer components` 디자인 시스템**(sa-tab/sa-chip/sa-card/sa-skeleton 등)이 100% 토큰 기반·터치타깃 44px 준수로 잘 설계됨.

### 1.2 약점

**(B-1) 토큰 우회 하드코딩이 광범위.**
프로젝트 단계 페이지·컴포넌트에서 하드코딩 Tailwind 색상 빈도(상위):
```
48 text-slate-400   33 bg-emerald-500   30 text-slate-900   28 text-slate-500
24 text-emerald-400 21 bg-amber-500     18 bg-red-500       11 bg-teal-500 ...
```
- 보정 레이어는 `text` 계열 일부만 커버. **`text-slate-900`(30건)은 다크 배경에서 거의 검정→안 보임**, `bg-*`/`border-*`는 토큰과 명도 정합이 깨질 수 있다.
- 의미색을 직접 hex/팔레트로 박아(`bg-emerald-500`=성공, `bg-amber-500`=경고, `bg-red-500`=오류) `--status-*` 토큰을 우회 → 라이트/다크 전환 시 명도 보장 안 됨.

**(B-2) 공유 컴포넌트 자체가 토큰을 우회한다.**
- `ModulePlaceholder.tsx`: `bg-indigo-500`, `text-indigo-500 dark:text-indigo-400`, `border-indigo-500/30` (24·36·52행) — 모든 단계 헤더에 박혀 있어 인디고가 토큰 외 "비공식 2차 강조색"으로 전 화면에 번짐.
- `legal/page.tsx`: 로딩 스피너 `border-teal-500`(18행) 하드코딩 — finance/bim은 다른 로딩, 단계별 로딩 스피너 색·형태 불일치.

**(B-3) 디스플레이 폰트가 일반체(Space Grotesk).**
`app/layout.tsx`·globals.css 모두 디스플레이 폰트로 **Space Grotesk** 사용. 본문 Noto Sans KR(국문 적합)은 OK지만, 디스플레이가 흔한 기하산세리프라 "AI 부동산 플랫폼"의 변별적 인상이 약하다. (※ 코드 내 폴백 체인 `var(--font-display), "Space Grotesk"`도 동일 폰트.)

**(B-4) 페이지별 모션·여백 편차.**
- `legal`: Framer Motion `initial/animate` 래핑 + `flex-col gap-12 pb-20`
- `finance`/`bim`: 모션 없음 + `grid gap-6`
- `cost`: 헤더·여백·CTA **전무**(10줄, `<BimCostDashboard>` 단독)
→ 같은 라이프사이클 안에서 진입 시 등장 애니메이션·섹션 간격·하단 여백이 단계마다 달라 "들쭉날쭉" 느낌.

## 2. 버튼명 / 용어 (라벨) 표준화

### 2.1 강점
- **사이드바 라벨이 일반인 친화적.** "90초 사업성 진단", "인허가 가능성", "시장·시세 분석", "투자 수익성 (ROI)" — 시행/시공 실무자가 바로 이해. 하위 항목 `└` 들여쓰기로 계층 표현(└ 등기부등본 열람, └ AI 시세추정 보고서).

### 2.2 약점

**(D-1) "분석 실행" 표기 난립.** grep상 동일 의미 버튼이 표기·간격이 제각각:
```
부지 분석 실행 / AI 심층 분석 실행 / 심층 분석 / 분석 실행 / AI 분석(30건) / 재실행 / 다시 실행 / 가상준공 생성 / 시작하기 / 생성하기
```
- "AI 분석" vs "AI 심층 분석" vs "심층 분석" — 사용자는 "심층"이 별도 유료/고급 기능인지 구분 불가.
- "재실행" vs "다시 실행" 혼용(동일 동작).

**(D-2) "로그인 실행".** 로그인 버튼에 "실행" 부착은 한국어로 어색(전 세계 표준은 그냥 "로그인"). 시스템 전반의 "~실행" 강박이 평범한 동작까지 침범.

**(D-3) 개발자 용어가 사용자 화면에 노출.**
- 단계 헤더 eyebrow `PROP / FINANCE`, `PROP / BIM` (영문 코드네임), 옆 `ko`(로케일 코드), `실연동`(=실데이터 연동, 내부 용어).
- finance 화면 "프로젝트 컨텍스트 / 프로젝트 ID: `1167cdda-9b1c-4698-b4f7-7c58526f28ac`" — **raw UUID 노출**.

**(D-4) 단계 수 카피 불일치.** 대시보드 "주소 검색 → 부지분석 → 결과 확인 → **나머지 6단계**" vs 실제 SSOT는 **10단계**(부지분석/법규/설계/BIM/시공계획/수지/금융/ESG/인허가/보고서). 또 상단탭은 **8그룹**으로 묶임. 6/8/10 세 숫자가 화면마다 다름.

**(D-5) 상단탭 라벨 vs 진행레일 라벨 미세 불일치.**
- 상단탭(`STAGE_GROUPS`): "입지 분석 / 법규 검토 / 건축 설계 / 사업성 검토 / 인허가·계약 / 시공 관리"
- 진행레일·SSOT(`STAGE_META`): "부지분석 / 법규검토 / 설계 / BIM / 시공계획 / 수지분석 / 금융분석 / ESG / 인허가 / 보고서"
→ "입지 분석"≠"부지분석", "건축 설계"≠"설계", "시공 관리"≠"시공계획". 같은 단계를 두 라벨로 부름.

### 2.3 용어 표준화 제안 (사전화)
| 현행(난립) | 표준안 | 규칙 |
|------|------|------|
| AI 분석 / AI 심층 분석 / 심층 분석 / 분석 실행 | **`{단계명} 분석`** (예: "부지 분석", "수지 분석") | 동사는 `분석`. "실행"·"AI"·"심층" 수식 제거(전 기능이 AI·심층임) |
| 재실행 / 다시 실행 | **`다시 분석`** | 동일 동작 단일화 |
| 생성하기 / 시작하기 / 가상준공 생성 | 결과물 명사 + `생성` (예: "보고서 생성", "3D 모델 생성") | "하기" 어미 제거 |
| 로그인 실행 | **`로그인`** | 자명한 동작에 "실행" 금지 |
| 입지 분석 / 부지분석 | **`부지 분석`** (전역 통일) | 탭·레일·SSOT 라벨 1:1 정합 |
| 건축 설계 / 설계 | **`설계`** | 〃 |
| 시공 관리 / 시공계획 | **`시공`** | 〃 |
| PROP / FINANCE, ko, 실연동 | (사용자 화면에서 삭제) | 내부 코드·로케일·연동상태는 사용자에게 무의미 |
| 프로젝트 ID: {uuid} | "프로젝트: {프로젝트명}" | UUID 비노출 |

> 실행 방법: `STAGE_META.label`과 `STAGE_GROUPS.label`을 단일 라벨로 합치고, 버튼 라벨은 i18n 사전(`i18n/dictionaries`)에 표준 키로 모아 컴포넌트가 사전만 참조하도록.

## 3. 화면구성 / 정보위계

### 3.1 표준 3구역 패턴 — 준수도 양호(라이브 확인)
라이브 finance/bim/overview 모두 일관된 구조 확인:
```
[컨텍스트 헤더]  서울 강남구 역삼동 1 / 일반상업지역   ← 단계 불문 고정 위치 ✓
[상단탭]        개요 입지 법규 설계 사업성 인허가 시공 보고서  ✓
[서브탭]        수지분석 금융분석 ESG (그룹 내 다단계만)  ✓
[Hero 헤더]     {단계명}. + 인용설명 + 우측 "범위 및 준비 상태" 패널
[위젯]          WorkspaceClient (실기능)
[다음 CTA]      NextStageCta (※ 라이브 미렌더 — §4 참조)
```

### 3.2 약점

**(C-1) Hero 헤더(ModulePlaceholder)가 "Coming Soon" 마케팅 블록처럼 보인다.**
- 제목을 `"{설명}"` 따옴표 인용문(에디토리얼 톤)으로 감싸고, 우측 패널 제목이 **"범위 및 준비 상태"** — 실제로 작동하는 페이지인데 "준비 상태"라는 단어가 미완성 인상을 준다.
- 컴포넌트 이름 자체가 `ModulePlaceholder`(=자리표시자). 라이브 운영 페이지의 메인 헤더로 쓰기엔 부적절. **화면 첫 스크롤(above the fold)을 실데이터가 아닌 정적 마케팅 카피가 점유** → 정보위계 역전.

**(C-2) 정보 중복·모순(컨텍스트 vs 입력 강제).**
- overview 화면: 상단 컨텍스트 헤더는 "서울 강남구 역삼동 1 / 일반상업지역"을 이미 표시하는데, 본문 메인은 **"프로젝트 주소를 입력하세요"** + 주소 검색창 → 이미 주소가 있는 프로젝트에 다시 입력을 요구(모순).
- finance 화면: "금융분석 입력 / 주소: 분석한 프로젝트 불러오기" 셀렉트 + "서울 강남구 역삼동 1" + "주소 변경" → **모듈마다 주소 재선택 UI** 반복(컨텍스트 자동흐름 미활용).

**(C-3) site-analysis 과밀.** 962줄·3D 디지털트윈·Leaflet 지도·다중 패널이 한 페이지에. 위 단계는 풍부한데 상대적으로 단계 간 정보밀도 격차가 크다(부지=과밀, 나머지=헤더+단일위젯).

### 3.3 빈상태·로딩·에러 — 부분 양호
- **빈상태 우수 사례:** finance 위젯 "수지분석(총사업비)이 아직 없습니다. 수지분석을 완료하면 개발금융(PF·이자·DSCR)이 자동 산출됩니다." → **의존 단계를 알려주는 dependency-aware 빈상태**(차별점).
- **로딩 불일치:** legal은 `border-teal-500` 스피너, site-analysis 동적 패널은 "…불러오는 중…" 텍스트 박스, 분양앱은 `sa-skeleton` 셔머. 3종 로딩 표현 혼재.
- 글로벌 `error.tsx` 존재(확인됨)하나 단계별 위젯 에러 폴백은 표준화 미확인.

## 4. 워크플로우 / 사용자 편의 — **치명 결함 발견**

### 4.1 (A) 진행 레일·다음단계 CTA가 라이브에서 死상태 — 최우선 수정
**증상(라이브 검증):**
- `LifecycleProgressRail`(완료/현재/다음추천/미시작 4상태 시각화, `{n}/10 · {pct}%`)이 **프로젝트 진입 중에도 화면에 없음**.
- `NextStageCta`("다음 단계: {단계명} 진입 ↗")도 **없음**.
- 라이브 localStorage: `propai-project-context = {"projectId":null, ...}` — **프로젝트 보는 중에도 컨텍스트 store의 projectId가 null**.

**근본 원인:**
- 두 컴포넌트 모두 `useProjectContextStore`의 `projectId`에 게이트(`if (!projectId) return null;`).
- `ProjectContextBinder`가 URL의 `id`를 `setProject()`로 바인딩해야 하는데, 라이브에서 `propai-project-context.projectId`가 채워지지 않음(바인딩 미작동 또는 async meta fetch 단계에서 실패 추정).
- 대조: 상단탭(`LifecycleNavigator`)은 **props로 projectId를 받음** → 정상 렌더(라이브 확인). store 의존 컴포넌트만 죽음.

**임팩트:** 부동산개발 SaaS의 핵심 차별화인 **"여정 가이드(어디까지 했고 다음은 뭐다)"가 통째로 보이지 않음.** 코드는 완성돼 있는데 켜지지 않은 상태.

> 권고: ProjectContextBinder의 바인딩이 실제 store 키에 반영되는지 1차 점검. 단기 안전장치로 레일/CTA가 **store 대신 URL projectId(props)도 폴백**으로 받게 하면 즉시 복구 가능(상단탭과 동일 전략).

### 4.2 단계 간 이동 — 다중 경로(과잉)
같은 단계로 가는 길이 ① 상단탭 ② (작동 시)진행레일 ③ (작동 시)다음CTA ④ 사이드바 일부 → 일관성은 좋으나, ②③이 죽어 현재는 상단탭/서브탭에 과의존. 상단탭(8그룹)과 서브탭(그룹 내 단계) 2단 구조는 직관적.

### 4.3 재입력 강제 vs 컨텍스트 흐름
- 컨텍스트 store·스냅샷 복원·dependency-aware 빈상태 등 **자동흐름 인프라는 갖춤**.
- 그러나 §C-2처럼 모듈마다 "주소 불러오기/변경" UI가 반복 노출돼 자동흐름의 이점이 화면상 가려짐. 컨텍스트가 이미 있으면 입력 UI를 접고(collapse) "변경" 링크만 노출하는 편이 깔끔.

### 4.4 모바일/반응형
- 상단탭 `overflow-x-auto scrollbar-hide` 가로 스크롤 — 모바일 대응 ✓(단 스크롤 어포던스(페이드) 없음, 분양앱 `sa-tabbar`에는 좌우 마스크 페이드 있는데 라이프사이클 탭엔 미적용).
- ModulePlaceholder `p-12 lg:p-20` — 모바일에서도 12 패딩이라 작은 화면에서 헤더가 세로로 길어 첫 화면을 다 잡아먹음(스크롤 비용 ↑).

## 5. 혁신 기회 (차별화 UX)

1. **단계별 완성도 게이지 + 잠금/추천 흐름.** 진행레일을 살린 뒤(§A), 각 단계 노드에 "필수 입력 N/M 완료" 미니 게이지를 달아 "지금 막힌 곳"을 한눈에. 의존 단계 미완 시 다음 단계를 흐리게+"부지분석 먼저" 툴팁(이미 빈상태에 의존정보 있음 → 노드 레벨로 승격).
2. **프로젝트 헬스 스코어(대시보드 요약).** 10단계 진행률·핵심 KPI(예상 ROI·인허가 가능성·총사업비 신뢰도)를 대시보드 단일 카드로 요약. 현재 대시보드는 마케팅 히어로 중심 → 로그인 사용자에겐 "내 프로젝트 현황 보드"가 더 가치.
3. **AI 인사이트 인라인 노출.** 각 단계 헤더의 "범위 및 준비 상태" 정적 패널을, 해당 프로젝트의 **AI 한 줄 인사이트**(예: "역삼동 일반상업, 용적률 잠재 800%까지 종상향 여지 — 법규 검토 권장")로 교체. 이미 interpreter 10종 보유 → 데이터는 있음.
4. **컨텍스트 칩 바.** 상단 컨텍스트 헤더를 주소뿐 아니라 면적·용도지역·용적률·예상ROI 칩으로 확장(모든 단계 공통), 한 번 분석한 값이 전 단계에서 따라다니는 "모세혈관" 가시화.
5. **가이디드 첫 실행(Empty→First Win).** 신규 프로젝트는 "주소 입력 → 부지분석 자동 → 30초 내 첫 인사이트"를 단일 풀스크린 스텝으로(현 6단계 온보딩 모달과 통합). "90초 사업성 진단" 메뉴와 연계.
